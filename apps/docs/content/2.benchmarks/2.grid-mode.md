---
title: Grid Mode Benchmark
description: Real-world viewport simulation — measure how fast a map viewport fills (not just single-tile latency)
---

Single-tile throughput numbers (req/s, p99) miss what users actually experience. When you pan or zoom a MapLibre GL JS map, the browser **fires a grid of tile requests in parallel** to fill the viewport. The metric that matters is **time-to-full-viewport** — how long until the last tile of the grid paints.

Grid mode simulates exactly this.

## Quick start

```bash
# Bring up the benchmark stack (tileserver-rs + tileserver-gl + martin + titiler)
docker compose -f benchmarks/docker-compose.yml --profile all up -d

# Run grid mode — default is 4×4 = 16 tiles per viewport
pnpm --filter @tileserver-rs/benchmarks run bench -- --mode grid

# Match a 1920×1080 browser viewport exactly
pnpm --filter @tileserver-rs/benchmarks run bench -- --mode grid --grid-size 5x4

# More iterations for tighter p99 confidence intervals
pnpm --filter @tileserver-rs/benchmarks run bench -- --mode grid --iterations 100
```

## How the grid is built

For a 1920×1080 viewport at 512 px tiles you need `⌈1920/512⌉ + 1 = 5` tiles wide and `⌈1080/512⌉ + 1 = 4` tiles tall — 20 tiles total. A 4×4 default captures a typical retina laptop viewport. The benchmark centers the grid on a well-known test location (Florence for PMTiles, Zurich for MBTiles, the Alps for STAC raster) and fires every tile request **concurrently via `Promise.all`** — identical to MapLibre's browser fetch pattern.

```
z=12 centered on Florence (2177, 1494), 4×4 grid:

  (2175, 1492)  (2176, 1492)  (2177, 1492)  (2178, 1492)
  (2175, 1493)  (2176, 1493)  (2177, 1493)  (2178, 1493)
  (2175, 1494)  (2176, 1494)  (2177, 1494)  (2178, 1494)
  (2175, 1495)  (2176, 1495)  (2177, 1495)  (2178, 1495)
```

## What gets measured

Each grid-load iteration records:

| Metric | What it represents |
| ------ | ------------------ |
| **Grid latency** | Time from firing the first request to receiving the last response (≈ time-to-viewport-full) |
| **Grids/sec** | Sustainable viewport loads per second (`1000 / avg_grid_latency_ms`) |
| **Throughput** | Total bytes transferred per grid (all 16-20 tiles) |
| **Max tile latency** | Slowest single tile in the grid — the one that blocks viewport paint |

Every zoom level gets **3 warmup iterations** (to prime caches, JIT, connection pools) before the measurement iterations kick in. This matters because cold-path STAC and COG tiles can take 30–60 s to hydrate, and warmup separates that from the steady-state number.

```text

Grid: 4×4 (16 tiles/viewport)
Iterations: 50 per zoom (+ 3 warmup)

📦 PMTILES Grid Benchmarks

  Testing tileserver-rs (PMTILES, 4×4 grid, 50 iterations)...
    z 8 (Italy       ) 4×4... 14.2 grids/s, 70ms avg, 98ms p99
    z12 (Florence    ) 4×4... 23.7 grids/s, 42ms avg, 61ms p99
    z14 (City Center ) 4×4... 22.9 grids/s, 44ms avg, 63ms p99

  Testing tileserver-gl (PMTILES, 4×4 grid, 50 iterations)...
    z12 (Florence    ) 4×4... 9.1 grids/s, 110ms avg, 189ms p99
```

The line that matters to a human: at z12 (city zoom, where users actually interact with the map), tileserver-rs fills a 16-tile viewport in **42 ms average** — roughly 2.5× faster than tileserver-gl's 110 ms. That's the difference between "instant" and "noticeable lag" on a pan.

## Why this matters more than req/s

Single-tile `req/s` numbers can mislead. A server that answers one tile in 10 ms but serializes the next 15 behind it still takes 160 ms to fill a viewport — the user experiences 160 ms, not 10 ms. Grid mode measures the parallel reality.

It also exposes **tail latency**. A server with 50 ms average and 600 ms p99 will feel janky because **every viewport load is only as fast as its slowest tile**. `maxTileLatency` in the per-iteration output surfaces this directly.

## CLI reference

```text
-m, --mode <mode>         single (default) | grid
-g, --grid-size <WxH>     grid dimensions, e.g. 4x4 (default), 5x4, 3x3
-i, --iterations <num>    per-zoom iterations (default: 50)
```

Raster formats automatically scale iteration count down by 3× (raster tiles are 10-50× slower than vector) so a full run completes in reasonable time without sacrificing statistical significance.

Grid mode supports the same `--type`, `--server`, and `--markdown` flags as single-tile mode — they all compose.

## When to use each mode

| Mode | Use when |
| ---- | -------- |
| `single` (default) | Measuring raw server capacity — how many individual tiles per second under sustained load |
| `grid` | Measuring user-perceived map load time — how fast a viewport fills |

Run both. Single-tile tells you whether your server can handle a load-test scenario (many clients, each fetching one tile). Grid tells you what a single user will experience sitting in front of the browser.
